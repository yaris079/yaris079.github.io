UVM学习笔记1--UVM世界观

2020-03-23

UVM世界观

1. 初识UVM

1.1 验证的作用

现代IC的前端设计通常是从一份需求说明书开始的。从需求说明书开始，IC工程师会把它细化为特性列表，验证工程师根据特性列表写出验证规格说明书。在验证规格说明书中，将会说明如何搭建验证平台，如何保证验证完备性，如何测试每一条特性，如何检测异常的情况等。设计人员使用Verilog(或VHDL)将特性列表转化成RTL代码，验证人员使用SystemVerilog设计验证平台，将RTL代码作为DUT(Design Under Test)进行验证。验证主要包括从特性列表到RTL转变的正确性，包括但不限于以下几点：

  • DUT的行为表现是否与特性列表中要求的一致。
  • DUT是否实现了所有特性列表中列出的特性。
  • DUT对于异常状况的反应是否与特性列表和设计规格说明书中的一致，如中断是否置起。
  • DUT是否足够稳健，能够从异常状态中恢复到正常的工作模式。

1.2 验证语言

SystemVerilog(以下简称为SV)是Verilog的一个扩展集，完全可以兼容Verilog。SV具有面向对象的语言特性，同时为验证提供了很多独有的特性，比如约束、功能覆盖率等。SV提供了API函数，可以快速将C/C++函数导入SV中，同时SV提供了自动的内存管理机制，对于生命周期结束的对象会自动释放，不需要担心内存泄漏的问题。

随着SV在验证语言层面的统一，UVM作为高级验证方法在2011年2月之后逐步融合，发布了UVM1.0。UVM面向所有数字设计，涵盖从模块级到芯片级，从ASIC到FPGA，从控制逻辑、数据通路到处理器的全部场景。

2. SystemVerilog特性回顾

在开始UVM之前，我觉得有必要回顾一下SV的语法，对于理解UVM还是很有帮助的。

2.1 接口

要完成充分的验证，需要给连接的各模块充分的激励。在发生激励之前，我们采用接口(interface)进行stimulator与DUT的连接。interface的基本作用主要有两点：

  • 简化模块间的连接
  • 实现类和模块之间的通信

interface可以看作是一束连线，如果将interface作为路由器，那么只要和interface接口一致，多个模块都可以使用这个interface，也不需要担心信号的驱动方向、连接性问题。

一种新加的和interface有关系的构造体是modport。由于接口中信号声明本身没有方向，为避免连接错误或反向驱动问题，为了限制信号的权限和访问方向，可以使用modport做进一步的说明。

interface里面可以带时钟、断言、任务(task)、函数(function)等定义。
一个interface 也可以有input,output或是inout端口。而modport提供了module的interface端口和在特定的module中控制task和function使用的方向性信息。这些端口的方向可以在module中可以看到。

总的来说,interface指明了TB和DUT之间的连接信号，clocking规定了信号之间的时序关系，而modport则明确了站在不同的角度对应信号的输入输出方向。通过虚拟接口的定义将以上内容封装起来。

2.2 激励

我们通过interface进行验证组件和DUT之间的连接，连接好后，激励通过接口向DUT传递，为了给出符合接口协议的激励，我们需要遵循接口的时序关系。

在stm_ini module里没有声明端口，激励不通过端口传递，而是interface进行数据的传递。virtual interface相当于interface的指针，可以被软件调用，而interface只能被硬件例化。virtual interface声明时，并不知道最终例化的regs_ini_if在何处，而它先假定regs_ini_if在仿真开始时可以被传递到virtual interface vif。

数据激励的产生可以构造task对stm_ini内的ts进行赋值，但此方法需要验证人员构思数据产生方法，且随机性不够。因此我们可以采用随机化激励的方法产生激励。

关于数据的随机化有以下几点需要注意：

  • 被标记为rand的成员变量作为随机成员可以通过randc轮转的方式完成随机化。即，在一个有效的随即周期里，每次生成的数据与之前生成的数不同，可用固定的周期遍历一次该变量可能的值域。
  • 随机成员要被随机化，必须由被例化的对象显式调用系统随机函数randomize()。
  • 悬空的成员类句柄会在随机过程中被忽略。
  • string和real无法被随机化

关于激励的随机化产生还有很多内容，包括约束块、随机化控制、随机化的稳定性等，这里就不再赘述。

2.3 Interface clocking

硬件世界和软件世界的连接可以通过interface来实现，同时可以在接口中声明clocking(时钟块)和采样的时钟信号，用来做信号的同步和采样。clocking块基于时钟周期对信号进行驱动或采样的方式，消除信号竞争问题。

2.4 组件间通信

Systemverilog中主要采用event、semaphore、mailbox三种方式实现组件间通信，具体实现方式与操作系统中通信方式类似。

3. UVM

SV作为验证的底层语言，自身已经可以搭建起一个简单的验证平台，但它撑不起芯片上庞大的验证规模，因此需要一个模板来让行文变得更容易，流畅和优美来应付各种复杂的场景。UVM更像是一个库，调用这个库的时候，可以使用其中写好的各种函数，大量代码可以复用，验证工程师只需要集中精力去设计所需的用例，考虑故障发生的情形就可以了。

验证使用于找到DUT中的bug，这个过程通常是把DUT放入一个验证平台中来实现的。一个验证平台要实现如下基本功能：

  • 验证平台要模拟DUT中各种真实使用情况，这意味着要给DUT施加各种激励，激励的功能是由driver来实现的。
  • 验证平台要能够根据DUT的输出来判断DUT的行为是否与预期相符合，完成这个功能的是记分板（scoreboard，也被称为checker）。
  • 验证平台要收集DUT的输出并把它们传递给scoreboard，完成这个功能的是monitor。
  • 验证平台要能够给出预期结果。在记分板中提到了判断的标准，判断的标准通常就是预期。当DUT运行时，验证平台也必须相应完成同样的过程。在验证平台中，完成这个过程的是参考模型（reference model）。
